# Memento v4.1.0 Session-Start Hotfix - 2026-05-18

## Incident

Expected behavior: the Codex PM memory MCP should be healthy before memory
dependent work so `mcp__memento__...` tools can be exposed when the app session
loads.

Actual behavior: the active session had no callable Memento MCP tools, while
`codex mcp get memento --json` showed the MCP configured and
`memento-mcp-runtime.ps1 status` initially reported `postgres_ready=True` and
`memento_health=False`. Running `memento-mcp-runtime.ps1 verify` started the
runtime and proved the direct MCP JSON-RPC path.

First mismatch/root cause: the SessionStart hook only instructed agents to use
Memento after tools were exposed. It did not check or start the HTTP runtime
when Memento was configured but down, so tool schema loading could happen before
the backend was healthy.

## Changes

- Merged upstream `origin/main`/`v4.1.0` into the local
  `%USERPROFILE%\.codex\tools\memento-mcp` checkout while preserving the Codex
  Windows runtime patches that keep in-process ONNX disabled and adaptive search
  parameter learning disabled for PM memory use.
- Aligned `package.json` and `package-lock.json` root versions to `4.1.0`.
- Confirmed the upstream GitHub release page marks `v4.1.0` as `Latest`
  as of 2026-05-18, and the local checkout reports `package.json` version
  `4.1.0`.
- Added `Invoke-MementoSessionStartEnsure` to the lightweight hook guard
  library and called it from `SessionStart`.
- Updated the Memento runtime status and workstation maintenance handoffs.
- Added a reusable incident pattern for "runtime healthy too late for MCP tool
  exposure."

## Verification Targets

- `memento-mcp-runtime.ps1 verify` must report required tools present,
  `context=pass`, `recall=pass`, and `tool_feedback=pass`.
- A synthetic `SessionStart` hook run must return context normally and must not
  schedule a start when Memento is already healthy.
- Active sessions may still require reload before `mcp__memento__...` tools are
  exposed. Direct HTTP JSON-RPC verification is fallback evidence only.

## Rollback

- For runtime rollback, stop Memento with
  `maintenance\scripts\memento-mcp-runtime.ps1 stop`.
- For source rollback, use the previous local Memento commit
  `3340f518dfb71c77e5e2a2910ab869fc1654b607` only after an explicit rollback
  request, because that discards the local v4.1.0 merge.
- For hook rollback, remove the `SessionStart` call to
  `Invoke-MementoSessionStartEnsure` and the helper function, then rerun hook
  smoke plus Memento verify.

## Residual Risk

The guard starts the runtime during `SessionStart`, but the current Codex app
may load MCP tool schemas before the hook returns. If a session still lacks
`mcp__memento__...` tools after the runtime is healthy, reload the app/session
and verify active tool exposure separately from backend health.
